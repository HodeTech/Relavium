## Context

The maintainer chose **all transports** for workstream 2.R (2026-06-26) â€” not just `stdio` â€” so the **network** MCP connections (`http` = the MCP *Streamable HTTP* transport, with legacy `sse` as its deprecated alias, and `websocket`; see [ADR-0052](0052-inbound-mcp-client-package-lifecycle-registration.md) Â§5) are in scope. [ADR-0034](0034-mcp-client-sdk-dependency.md) guardrail 2 and [mcp-integration.md](../reference/shared-core/mcp-integration.md) require an MCP `url` to pass the **one shared SSRF range-block primitive** (the same used for provider base URLs, the `http_request` tool, and media egress) and to honor an **"explicit local-endpoint opt-in"** â€” but the pre-implementation doc review (2026-06-26) verified that **neither** is owned by an ADR:

1. **The enforcement model for a socket the SDK opens is undecided.** [ADR-0043](0043-media-egress-failover-rematerialization-ssrf.md) landed **connect-by-validated-IP** for media HTTPS bytes (the engine owns the policy, the host resolves DNS and connects to the *validated IP*, and re-validates on every redirect hop) â€” closing the DNS-rebind window. But the MCP SDK's `StreamableHTTP`/`SSE`/`WebSocket` transports open their **own** socket, so that guarantee does **not** automatically extend: a naive construction-time literal check on the `url` leaves exactly the rebind window ADR-0043 was built to close. The schemas (`McpServerRefSchema`/`McpServerRegistrationSchema`) today validate only the **scheme** and **embedded-credentials** at parse â€” correctly, since the range-block is a runtime/host concern â€” so the runtime enforcement point is a hole.
2. **The local-endpoint opt-in surface does not exist.** [mcp-integration.md](../reference/shared-core/mcp-integration.md)'s own canonical example is `http://localhost:4000/mcp`, which the same doc says "require[s] that explicit opt-in" â€” but no authored field or config flag implements it. The **media seam deferred the identical opt-in "behind a fresh ADR"** (the `MEDIA_URL_SOURCE_ENABLED` flag note in [`content.ts`](../../packages/shared/src/content.ts), `SEC-EGRESS-3` in [`media-egress.ts`](../../packages/db/src/media-egress.ts):312-314), so this is a genuinely shared, still-open security surface, not an MCP-only nicety â€” and `SEC-EGRESS-3` explicitly mandates that any wiring of the `allowPrivate` opt-in **must add an explicit port-allow-list decision** (a crafted `https://host:22/` to a permitted-private address would otherwise be reachable).

Getting either wrong is a real SSRF exposure (a malicious or compromised MCP registry pointing a "remote" server at `169.254.169.254` or a loopback admin port), so both warrant a recorded decision rather than an implementer's guess â€” and, per the `0042`â†”`0043` precedent (substrate vs. egress-security), they belong in a security ADR separate from the structural [ADR-0052](0052-inbound-mcp-client-package-lifecycle-registration.md).

## Decision

**We record the network-transport MCP egress security contract below, reusing the one shared SSRF primitive and providing the per-server local-endpoint opt-in surface that MCP needs now and that media can later adopt.** `stdio` is out of scope (it carries no `url`; its only egress concern is the explicit-env spawn pinned in [ADR-0052](0052-inbound-mcp-client-package-lifecycle-registration.md) Â§6 / [ADR-0034](0034-mcp-client-sdk-dependency.md) g5). Each network mechanism below is security-sensitive and gets a **dedicated security-review pass when it lands**, exactly as [ADR-0043](0043-media-egress-failover-rematerialization-ssrf.md) gated its egress mechanisms.

### 1. One shared SSRF primitive, never a second parser

Every `http`/`websocket` `url` is validated through the **existing** `@relavium/shared` range-block primitive (`isPrivateOrLocalHost`, [`content.ts`](../../packages/shared/src/content.ts)) â€” the private/loopback/link-local/`169.254.169.254`/CGNAT/IPv4-mapped-IPv6 ranges are blocked, **never a hand-rolled second check** ([ADR-0029](0029-tool-policy-hardening.md) / [ADR-0034](0034-mcp-client-sdk-dependency.md) g2). Two host-extraction details are pinned, because `extractHttpsHost` is **HTTPS-only**: (a) the range-block runs on a **transport-agnostic host extraction** that handles `wss`/`ws` and `http(s)` authorities (reusing `isPrivateOrLocalHost` on the extracted host, not a second parser); (b) the **remote-endpoint-must-be-`https`/`wss`** rule is enforced **at the host connect check, not at parse** â€” the parse-level `SAFE_MCP_URL` deliberately admits plaintext `http`/`ws` (it gates scheme-shape only), so plaintext is permitted **only** for an `allow_local_endpoint` server (local dev), and a **remote** plaintext `http`/`ws` endpoint is **rejected at connect**. (No-embedded-credentials is already enforced at parse.)

### 2. Connect-by-validated-IP is the target; pre-connect host validation is the floor; redirects re-validate

The **target** enforcement is **connect-by-validated-IP**, reusing the [ADR-0043](0043-media-egress-failover-rematerialization-ssrf.md) host-egress mechanism (resolve DNS â†’ validate the resolved IP against the primitive â†’ connect to *that* IP), applied through a **custom dialer/transport the SDK accepts** (its Streamable-HTTP/WebSocket transports take an injectable `fetch`/agent). For the HTTP layer, **every redirect hop re-runs the primitive** on the resolved target, so a DNS-rebind or a redirect-to-private is blocked mid-connect, not only at construction. Unlike media's **one-shot** fetch ([ADR-0043](0043-media-egress-failover-rematerialization-ssrf.md)), `http`/`websocket` MCP are **persistent** connections: connect-by-validated-IP pins the **initial dial and every reconnect** to the validated IP, and once connected the bytes flow only to that pinned address â€” a long-lived session cannot be re-pointed to a private IP without a fresh, re-validated dial.

Where the SDK transport does **not** expose a hook to pin the dialer, the enforced **floor** is **pre-connect host validation** on the resolved `url` plus a **documented residual DNS-rebind window**, and a **forward obligation** (logged in `deferred-tasks.md`) to land the pinned dialer once the transport permits. The host owns the validated connect; the engine supplies the policy. *(Considered: pre-connect validation as the permanent bar â€” rejected as the *target*: it accepts a rebind window indefinitely on a connection that can carry credentials; a fully hand-rolled MCP transport that we fully control the socket of â€” rejected: it re-implements the SDK protocol, the exact anti-goal of [ADR-0034](0034-mcp-client-sdk-dependency.md).)*

### 3. The local-endpoint opt-in: a per-server authored field, scoped to the declared `host:port`

A **per-server boolean `allow_local_endpoint`** on `McpServerRefSchema` **and** `McpServerRegistrationSchema` wires to the host SSRF `allowPrivate` **for that server only**. Absent â‡’ a private-range `url` is **rejected** (the secure default; the canonical `http://localhost:4000` example must set it). The opt-in is **scoped to the exact declared `host:port`** of that server's `url` â€” **not** any port on that host. This is the **port-allow-list decision `SEC-EGRESS-3` mandates** ([`media-egress.ts`](../../packages/db/src/media-egress.ts):312-314) for any wiring of the `allowPrivate` opt-in: without it, `allow_local_endpoint` on a `localhost:4000` server would let a crafted `url` or a redirect reach `localhost:6379` (Redis), `:5432` (Postgres), `:22` (SSH), or the Docker socket on the same permitted-private host. With it, the validated connect (and every redirect re-validation, Â§2) must target the declared `host:port`; any other private `host:port` stays blocked. For that **exact declared `host:port`** the flag relaxes the private-range block **and permits plaintext `http`/`ws`** (a local-dev MCP server is typically plaintext, which is why the canonical `http://localhost:4000` example needs the opt-in) â€” it **never** relaxes the no-embedded-credentials check, and **never** permits plaintext for a *remote* endpoint (a remote `url` still must be `https`/`wss`, enforced at connect per Â§1). It is the **one reusable opt-in shape** the media seam (which deferred the same decision) can later adopt. The `allow_local_endpoint` field is **added to both schemas + the specs at 2.R**. *(Considered: relax to any port on the permitted host â€” rejected: that is exactly the `SEC-EGRESS-3` hole; a single global `allowPrivate` config flag â€” rejected: not least-privilege; no opt-in at all â€” rejected: the documented local-dev MCP use-case needs it.)*

## Consequences

### Positive

- The network-MCP SSRF surface is closed on the **one** shared primitive with a concrete enforcement model (connect-by-validated-IP target, pre-connect floor, per-hop redirect re-validation) â€” no second parser, no silent rebind hole left to an implementer's guess.
- The local-endpoint opt-in is least-privilege (per-server) and is deliberately the **shared** surface media can adopt, resolving the "behind a fresh ADR" deferral the media seam left open â€” one opt-in concept across the product.
- The secure default holds: a private-range MCP `url` is rejected unless the author explicitly opts that one server in, so a malicious registry entry cannot reach a metadata endpoint or a loopback admin port by default.

### Negative

- Connect-by-validated-IP through the SDK's transports depends on the SDK exposing a dialer/`fetch` hook; where it does not, 2.R ships the pre-connect floor with a documented residual rebind window and a tracked hardening obligation â€” an honest, bounded gap, not a silent one (mitigated: each network mechanism gets a dedicated security-review pass at landing, per the ADR-0043 precedent; the floor still blocks the literal-private and redirect-to-private cases).
- A new authored field (`allow_local_endpoint`) on two shared schemas is a public-contract change touching `@relavium/shared` (mitigated: additive, strict-validated, defaults to the secure-closed behavior, and is the surface media will reuse rather than a throwaway).

### Neutral

- This ADR owns network-transport MCP egress security; the structural decisions (package, lifecycle, registration, validator, the reference linkage, named-secret resolution) live in [ADR-0052](0052-inbound-mcp-client-package-lifecycle-registration.md). `stdio` security is the explicit-env spawn in ADR-0052 Â§6.
- When the media seam adopts `allow_local_endpoint`, that is an amendment recorded against [ADR-0043](0043-media-egress-failover-rematerialization-ssrf.md)/this ADR, not a silent reuse.
