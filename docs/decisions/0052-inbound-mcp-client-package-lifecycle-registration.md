# ADR-0052: Inbound MCP client — the `@relavium/mcp` package boundary, the host-injected connection lifecycle, host-side tool registration, a dependency-free schema→validator compiler, and the agent↔config reference linkage (implements ADR-0034)

- **Date**: 2026-06-26

## Context

[ADR-0034](0034-mcp-client-sdk-dependency.md) settled **which** MCP client (the official `@modelcontextprotocol/sdk`) and **when** (workstream 2.R, off the M3 critical path), with five binding guardrails — but it deliberately did **not** settle the implementation *shape*. A pre-implementation documentation review (2026-06-26, a five-dimension verified sweep) confirmed the registration/dispatch/event **spine already exists** — the `mcp_call` built-in + its `McpCapability` host seam ([`builtins.ts`](../../packages/core/src/tools/builtins.ts), [`types.ts`](../../packages/core/src/tools/types.ts)), the config `[[mcp_servers]]` global→project merge ([`resolve.ts`](../../apps/cli/src/config/resolve.ts)), the `ToolDef` contract, and the `agent:tool_call`/`agent:tool_result` events reused verbatim — but surfaced a cluster of **unowned decisions** that, guessed ad hoc inside a feature PR, would violate a non-negotiable (engine purity, SDK-confinement, no-new-dependency-without-an-ADR) or leave the canonical spec promising a behavior no schema implements.

The maintainer settled the four load-bearing forks (2026-06-26): **all three transports** (`stdio` + `sse` + `websocket`) are in 2.R scope; a **dedicated package** owns the layer; the **by-name reference form** lands now; and the JSON-Schema→validator compiler is **dependency-free**. The binding constraints those choices sit against: `packages/core` stays platform-free ([CLAUDE.md](../../CLAUDE.md) rule 5) yet the SDK's `stdio` transport pulls `node:child_process`; SDK types must be confined like provider SDKs ([ADR-0034](0034-mcp-client-sdk-dependency.md) g3 / [ADR-0011](0011-internal-llm-abstraction.md)); no new runtime dependency may be adopted without an ADR (rule 2); and [ADR-0037](0037-engine-tool-execution-boundary.md) already frames MCP ToolDefs as additive registration, not a registry reversal. These are exactly the cross-cutting, guarantee-bearing choices the project records as an ADR before code, so a feature PR neither re-derives them nor quietly weakens a guarantee.

## Decision

**We implement the inbound MCP client as the pinned choices below — implementing (not superseding) [ADR-0034](0034-mcp-client-sdk-dependency.md).** The network-transport egress security half (SSRF on the shared primitive + the local-endpoint opt-in) is the sibling [ADR-0053](0053-mcp-network-transport-egress-security.md); `stdio` carries no `url` and its only egress concern is the explicit-env spawn pinned below. The pinned SDK version lives in [tech-stack.md](../tech-stack.md) + the pnpm `catalog:` when the package lands, not here.

### 1. A new `@relavium/mcp` package owns the mechanism and fences the SDK

The MCP integration layer is a **new workspace package, `@relavium/mcp`**, mirroring the `@relavium/llm` seam: it confines `@modelcontextprotocol/sdk` **and** `node:child_process`, and exports outward **only Relavium/Zod shapes** — a discovered-tool→`ToolDef[]` producer plus a concrete `McpCapability` implementation (the host arm of the engine's existing seam). It is consumed by the **Node hosts** (`apps/cli` now, `apps/vscode-extension` in Phase 4); the **desktop's Rust backend** owns its own stdio lifecycle ([mcp-integration.md](../reference/shared-core/mcp-integration.md)) and does not import it. **`packages/core` never imports it** — the engine continues to see only `ToolRegistry`/`McpCapability` shapes, so engine purity (rule 5) and SDK-confinement ([ADR-0034](0034-mcp-client-sdk-dependency.md) g3) both hold. *(Considered: a module inside `apps/cli` — rejected: VS Code would re-implement it and the SDK leaks into the app surface; inside `packages/core` — rejected outright: `node:child_process` + SDK types break platform-purity and the seam rule.)*

### 2. A host-injected `McpClientManager` owns connection lifecycle; degraded init fails loud

The host constructs an **`McpClientManager`** (scoped per run / per chat session), establishes each declared server's connection at agent/run start, keeps it alive for the run duration, and tears it down on the run's **terminal event** — the platform-free engine never spawns or connects, exactly as [ADR-0036](0036-run-loop-substrate-event-bus-and-execution-host.md)/[ADR-0018](0018-desktop-execution-and-rust-egress.md) delegate I/O to the host. This is the lifecycle narrative [mcp-integration.md](../reference/shared-core/mcp-integration.md) promised but never wrote; it lands in `shared-core-engine.md` and the spec repoints to it.

Degraded init is **fail-loud**: a declared server that will not spawn, will not connect, or fails `tools/list` **fails the agent at init** with a typed, secret-free error — no silent capability loss, consistent with the project's fail-closed posture (`media_store_unavailable`, keychain fail-closed). A **mid-run** server crash or a `tools/call` transport error maps onto the existing `ToolExecutionError` → retryable `tool_failed` (the node retry budget governs); a **tool-reported** error returns a correctable `isError` tool_result the agent loop can recover from. Restart-on-crash is a desktop-Rust concern, out of 2.R. *(Considered: degrade-gracefully — register no tools and continue — rejected: it silently drops an explicitly-declared capability, the opposite of secure-by-default.)*

**The `stdio`-spawn trust boundary.** Spawning a declared `stdio` server runs a local binary or an `npx`-installed package — an **arbitrary-code-execution + supply-chain** surface categorically unlike a network MCP connection (which talks to an *already-running* server). For 2.R the trust model is **author trust**: a server declared in the user's *own* committed config/agent YAML is author intent, spawned via the dedicated explicit-env/`shell:false` path (§6) — **not** routed through `run_command`'s model-chosen `allowedCommands` allowlist ([ADR-0029](0029-tool-policy-hardening.md)), because an author-declared config command and a *model-chosen* command are different trust classes. The **imported/shared untrusted workflow** case (a `.relavium.yaml` from an untrusted source declaring `command: npx, args: ['-y', 'malicious-pkg']`) and the **`npx` auto-install** convenience ([mcp-integration.md](../reference/shared-core/mcp-integration.md)) are an arbitrary-code-execution surface **out of 2.R's baseline scope** — 2.R's threat model is the author's own local files; an import-trust/consent gate plus dependency pinning are a **forward obligation** (`deferred-tasks.md`).

### 3. Host-side static `ToolDef` assembly — no engine registration API

Discovery runs at **agent startup**; the host composes `[...BUILTIN_TOOLS, ...mcpDefs]` and passes the array into the **existing** `createToolRegistry({ tools })`, and the same defs back the `McpCapability` routing. The `ToolRegistry` stays a **static set** (no `register()`/`add()` mutation method); `resolveGrant`/`buildLlmTools` already work for any `ToolDef`. This is the **additive host-side assembly** that [tool-registry.md](../reference/shared-core/tool-registry.md) anticipates (its "MCP/plugin ToolDefs register dynamically (2.R)" wording — reconciled at 2.R, §4) and [ADR-0037](0037-engine-tool-execution-boundary.md) frames as the `ToolHost` surface "grow[ing] as MCP (2.R) … land[s] — an intentional, additive evolution … not a reversal" — **not** a registry-mutation reversal. The agent's `tools:` grant **and** the per-server `tools_allowlist` narrow which discovered tools become granted defs ([ADR-0029](0029-tool-policy-hardening.md) narrow-only); the `mcp_{server}_{tool}` namespace disambiguates same-named tools across servers. **Discovery runs per run/process**; the durable cross-invocation tool-list cache ([mcp-integration.md](../reference/shared-core/mcp-integration.md), ~1h per-`(command,args)`) is a **deferred optimization**, not 2.R (it re-runs each process, and a transport-covering cache key is a tracked follow-up). *(Considered: an engine-side dynamic-registration API — rejected: it breaks the static-registry invariant and leaks connection lifecycle into the platform-free engine.)*

### 4. The LLM-visible id is `mcp_{server}_{tool}`; routing uses a discovery map, not a name split; a hardened dependency-free compiler builds the validator

The **namespaced, LLM-visible tool name is `mcp_{server}_{tool}`** (underscore) — LLM tool-name charsets reject `:`, so the colon `mcp:<server>:<tool>` example **and** the "register dynamically" wording in [tool-registry.md](../reference/shared-core/tool-registry.md) + [`types.ts`](../../packages/core/src/tools/types.ts) are **reconciled at 2.R** to the underscore form + "host-side assembled." **Routing never parses the name back** into `(server, tool)`: the host builds an explicit `namespacedId → { server, tool }` map at discovery (it already knows each tool's origin), so a tool name that itself contains `_` is never ambiguous — the server segment is the agent ref's kebab `id` (or, for a `ref`, a **sanitized** form of the registration `name`, §5), used for *display* only. The namespaced name is **validated against the LLM tool-name charset**, and a post-sanitization **name collision across servers fails closed at discovery** (it never silently shadows another tool).

A **dependency-free, in-house compiler** turns the server-reported JSON Schema (the subset MCP servers emit in practice — `object`/`string`/`number`/`integer`/`boolean`/`array`/`enum`/`const`/`required`/nullable + nesting) into the executable `ToolDef.parseArgs` validator that gates every dispatch. This gate is **ours to build**: the doc review verified the MCP SDK does **not** validate call arguments against the server-reported `inputSchema` (it returns the raw schema and forwards calls), while [tool-registry.md](../reference/shared-core/tool-registry.md) requires an *executable* validator, not a bare schema. The input schema is **server-controlled and in-scope untrusted** (the threat model includes a malicious/compromised MCP server), so the compiler is **adversarially hardened** — bounded nesting depth + node count, no unbounded/backtracking regex (no ReDoS), and **fuzz/adversarial-tested**, not only checked against benign real-server schemas. It is a rule-2 **commodity** (a schema-shape→validator transform), **not** a rule-3 security primitive (it hand-rolls no crypto/TLS/keychain); its security property is that it **fails closed** — an unsupported or malformed construct **rejects the tool at discovery**, never admits it unvalidated. **No new runtime dependency** (rule 2 / [ADR-0034](0034-mcp-client-sdk-dependency.md)). *(Considered: a vetted validator dependency — `ajv` / `json-schema-to-zod` — rejected per the maintainer's build-in-house choice; recorded so a future need for full JSON-Schema coverage re-opens it via an [ADR-0034](0034-mcp-client-sdk-dependency.md) amendment, never an ad-hoc adoption inside a PR.)*

### 5. The agent↔config linkage: a lean by-name reference form, mutually exclusive with inline fields; one reconciled transport vocabulary

We add a **reference branch** to `McpServerRefSchema`: an agent entry may be `{ ref: <registration-name>, tools_allowlist? }`, resolved against the merged config `[[mcp_servers]]` registrations — realizing [mcp-integration.md](../reference/shared-core/mcp-integration.md)'s "register once, reference from many agents" promise that **no schema currently implements**. **Invariants** (enforced by the schema's existing `superRefine`): `ref` is **mutually exclusive** with every inline connection field (`transport`/`command`/`url`/`env`) — present `ref` ⇒ identity and connection come from the *registration*, and the inline `id`/connection fields are absent; the LLM-namespace server segment derives from a **sanitized** form of the registration `name` (a free `nonEmptyString`), and routing uses the discovery map, never a name split (§4). The inline self-contained form stays valid and unchanged (the 2.R acceptance fixture uses it).

**Transport vocabulary (reconciled to the current MCP spec).** Both schemas converge on **`stdio | http | websocket`**, where **`http` denotes the MCP *Streamable HTTP* transport** — the current spec-standard network transport (the SDK's `StreamableHTTPClientTransport`). This makes the config schema's existing `http` **correct and current** (not a pre-standard placeholder), and the agent schema's `sse` becomes a **deprecated alias** of `http` (the SDK's legacy HTTP+SSE transport, accepted for older servers). The url-scheme validation is unchanged (`http` → `https?`, `websocket` → `wss?`). The agent-yaml-spec, config-spec, and mcp-integration.md are **updated at 2.R** to this one vocabulary. *(Considered: `stdio|sse|websocket` per the legacy doc — rejected: SSE is deprecated by the MCP spec in favor of Streamable HTTP, and [ADR-0053](0053-mcp-network-transport-egress-security.md) names the SDK's StreamableHTTP transport, so the lean current set is `stdio|http|websocket` with `sse` as a legacy alias; keep agents inline-only and delete the register-once promise — rejected per the maintainer's choice, so a referenced registration's `url`-guarding + env-secret handling apply consistently whether a server is inlined or referenced.)*

### 6. Named-secret resolution into an explicit child environment — in an isolated keychain namespace

`{{secrets.<name>}}` inside a server's `env` resolves from the **OS keychain** via a named-secret account scheme that is **namespace-isolated from provider keys**: the account is **`mcp-secret:<name>`** (`service = relavium`), a **distinct namespace** from the provider-key accounts (`{providerId}:{keyId}`, e.g. `anthropic:default`) and the `search-provider` account ([keychain-and-secrets.md](../reference/desktop/keychain-and-secrets.md) §Entry naming). **`{{secrets.<name>}}` resolves *only* within the `mcp-secret:*` namespace** — it can **never** reach a provider-key or search-provider account. This is load-bearing under the "a shared/imported workflow is the invite" distribution model: without the isolation, a hostile `env: { LEAK: '{{secrets.anthropic}}' }` in an imported agent could exfiltrate the provider key into a malicious MCP server's environment; the isolated namespace closes that path. A `RELAVIUM_MCP_<NAME>`-style **env-var fallback** (also namespace-scoped) mirrors the provider keychain→env→error chain ([ADR-0019](0019-cli-node-keychain-library.md)). The resolved value is injected **only** into the **explicitly-constructed child environment** at spawn (the declared `env` plus a minimal base, reusing the existing explicit-env/`shell:false` hygiene — never a blanket copy of the host environment, [ADR-0034](0034-mcp-client-sdk-dependency.md) g5) and **never** appears in a committed YAML, a log line, an event payload, or the `--json` stream. A missing secret is **fail-closed** (the server fails to spawn with a secret-free error). The `mcp-secret:` account scheme is **added to [keychain-and-secrets.md](../reference/desktop/keychain-and-secrets.md) at 2.R**, under [ADR-0006](0006-os-keychain-for-api-keys.md)/[ADR-0034](0034-mcp-client-sdk-dependency.md). This is a **new secret-custody surface**; a generalized `SecretResolver` is a Phase-3 forward obligation, not built here.

## Consequences

### Positive

- The four ADR-0034-deferred shape decisions get an owner before any code is written: a feature PR neither re-derives them nor risks an ad-hoc weakening of engine purity, SDK-confinement, or the no-new-dependency rule.
- The SDK + `node:child_process` are fenced in one package that surfaces only Relavium types, exactly mirroring `@relavium/llm` — the engine stays platform-free and the same layer is reused by CLI and VS Code (the desktop keeps its Rust path).
- Host-side static assembly realizes ADR-0037's promise with **zero** engine-interface change (the registry, grant resolution, and LLM-tool surfacing already accept any `ToolDef`), so MCP is genuinely additive behind the existing abstraction.
- The validator gate is dependency-free and fail-closed, honoring build-in-house + no-new-dep while still enforcing schema-validate-before-dispatch on tools whose schemas the SDK does not check.
- The by-name reference form makes the canonical spec honest (the "register once" story now has a schema), and named-secret resolution gives MCP servers credentials without a secret ever touching a committed file, a log, or an event.

### Negative

- A new workspace package is real scaffolding (package.json, tsconfig, ESLint/Vitest wiring, the SDK pin) and a new seam to keep clean — mitigated: it follows the established `add-package` scaffold and the `@relavium/llm` confinement pattern exactly.
- The in-house JSON-Schema→validator compiler is code we own and must test against real-server schemas, and it deliberately covers only a subset — mitigated: the subset is what MCP servers emit in practice, unsupported constructs fail closed (never silently pass), and the escape hatch (a vetted dep via an ADR-0034 amendment) is recorded.
- The by-name reference form is a shared-schema change (`McpServerRefSchema` + the transport-vocabulary reconciliation in `McpServerRegistrationSchema`) touching `@relavium/shared`, a public contract — mitigated: it is additive (the inline form is unchanged), strict-validated, and reconciles two schemas that were already drifting.
- Named-secret resolution is a new secret-custody surface, and the spawned child is a (possibly hostile) MCP server — the live exfil concern is an imported workflow naming a *provider* key — mitigated: the `mcp-secret:*` keychain namespace is **isolated** from provider-key/search-provider accounts so `{{secrets.*}}` can never resolve a provider key; injection is into the explicit child env only; a miss is fail-closed; the value is asserted never to reach a log/event/`--json` line; the generalized resolver is deferred, not improvised.
- A declared `stdio` server spawn is arbitrary local-code / `npx` execution — acceptable under 2.R's author-owns-the-files threat model (author trust, the dedicated explicit-env spawn, not the model-chosen `run_command` allowlist), but the imported-untrusted-workflow + `npx` auto-install supply-chain surface is real — mitigated: it is explicitly out of 2.R baseline scope with an import-consent-gate + dependency-pinning forward obligation logged in `deferred-tasks.md`.

### Neutral

- This ADR is the authoritative record for the inbound MCP client's structure; ADR-0034 remains the record for the dependency-and-slot decision. A future reader implementing or extending MCP should read [ADR-0034](0034-mcp-client-sdk-dependency.md), this ADR, and [ADR-0053](0053-mcp-network-transport-egress-security.md) together.
- The outbound direction (workflow-as-MCP-server) is untouched and remains a later workstream, exactly as ADR-0034 scoped it.
