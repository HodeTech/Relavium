# ADR-0043: Media egress — the binary media-egress capability, the `FallbackChain`↔`MediaStore` re-materialization contract, and the SSRF mechanism half (amends ADR-0031)

- **Status**: Accepted
- **Date**: 2026-06-18
- **Related**: [0031-llm-seam-shape-amendment-multimodal-io.md](0031-llm-seam-shape-amendment-multimodal-io.md) (**this ADR amends it** — append-only, ADR-0031 is unchanged in history; it resolves ADR-0031's sidecar-ownership self-contradiction, lands the `url`-carrier flag-on with the completed SSRF primitive, and pins the engine-resolves-before-egress split), [0011-internal-llm-abstraction.md](0011-internal-llm-abstraction.md) (the seam-purity rule — no vendor type, no `MediaStore` held by an adapter, the chain stays byte-free), [0030-llm-seam-shape-amendment-reasoning-response-format-provider-executed.md](0030-llm-seam-shape-amendment-reasoning-response-format-provider-executed.md) + [0039-same-provider-reasoning-replay.md](0039-same-provider-reasoning-replay.md) (the strip-on-failover / `beginEntry` reasoning-signature precedent this mirrors), [0038-agentrunner-llm-call-boundary.md](0038-agentrunner-llm-call-boundary.md) (the per-node `FallbackChain` + host-injected provider resolution this hooks), [0029-tool-policy-hardening.md](0029-tool-policy-hardening.md) (the one shared SSRF range-primitive + egress discipline), [0042-engine-media-storage-substrate-mediastore-deinline-retention.md](0042-engine-media-storage-substrate-mediastore-deinline-retention.md) (the `MediaStore`/`deInlineMedia` substrate this consumes), [0018-desktop-execution-and-rust-egress.md](0018-desktop-execution-and-rust-egress.md) (the host-delegated egress pattern the binary capability follows), [../standards/security-review.md](../standards/security-review.md) (the SSRF rule + the one range-primitive), [../reference/shared-core/llm-provider-seam.md](../reference/shared-core/llm-provider-seam.md), [../analysis/multimodal-io-design-2026-06-07.md](../analysis/multimodal-io-design-2026-06-07.md) (decisions A7, B1, B5).

## Context

[ADR-0031](0031-llm-seam-shape-amendment-multimodal-io.md) made media a first-class seam value and bound the egress-side obligations, but left three of them **shape-without-mechanism, deferred to 1.AF**, and one of them **internally contradictory**:

1. **The `FallbackChain`↔`MediaStore` re-materialization (B5) is self-contradictory in ADR-0031.** ADR-0031 says the provider-hosted ephemeral ref (Gemini `fileUri`, OpenAI `file_id`/`audio.id`) lives "**only** in a **process-scoped adapter sidecar** keyed by `(provider, sha256)`" *and* that it is "stripped and re-materialized from the canonical handle on `FallbackChain` failover," with the strip/re-materialize "owned by the 1.K `FallbackChain`" (Consequences). But adapters are mandated **stateless `string → string`** that never hold a `MediaStore`. So *who* holds the sidecar, and *how* a seam-pure, byte-free chain re-materializes bytes without taking a host/`MediaStore` dependency, is unresolved — and guessing risks the retried request **racing a half-uploaded ref** or a foreign provider ref leaking across the boundary (an I1 vendor-type leak).

2. **`EgressCapability.fetch` cannot carry binary media.** Verified at [`tools/types.ts`](../../packages/core/src/tools/types.ts): `EgressResponse.body` is **`string`-typed**. A string body silently corrupts binary image/audio/video bytes. ADR-0031's SSRF guardrail says "the engine fetches and re-hosts [a provider output URL] to a handle inside the `deInlineMedia` pass," but there is **no byte-shaped egress path** for it to use.

3. **The SSRF mechanism half is deferred and its 1.AF scope was contradictory across docs.** 1.AE landed the **policy** half (the one shared range-primitive: HTTPS-only, no-creds-in-URL, private/loopback/link-local/`169.254.169.254`/CGNAT/IPv4-mapped-IPv6 literal checks). The **mechanism** half — DNS-resolve + connect-by-validated-IP + per-hop redirect re-validation — was deferred, and [deferred-tasks.md](../roadmap/deferred-tasks.md) framed it as a Phase-2 obligation while the 1.AE note deferred it "to 1.AF." The seam's `url`-acceptance flag (`MEDIA_URL_SOURCE_ENABLED`) is **already ON** — 1.AE flipped it to `true` with the policy half (`content.ts`), so input URLs are SSRF-policy-validated and **accepted at ingest** today. What is still missing is the **behavior**: nothing yet *fetches and re-hosts* a URL to a handle, because the mechanism half is unbuilt. **The maintainer has decided (2026-06-18) to land the mechanism half in 1.AF** (rather than ship 1.AF with the obligation open) — so this ADR records the mechanism placement and the now-live output-URL fetch/re-host **behavior**, not a flag flip (the flag is already on).

4. **The adapter handle/url resolution split is ambiguous.** ADR-0031's body says the **engine** resolves handles via `MediaStore.resolveForEgress` *before* egress and "adapters never hold a `MediaStore`," yet the 1.AE adapter code carries `// resolved at egress by MediaStore (1.AF)` accept-but-skip comments that imply *adapter-side* wiring. The split must be pinned so the adapters stay pure.

These are cross-seam decisions — they sit exactly on the I1 boundary ADR-0011 protects and parallel the [ADR-0038](0038-agentrunner-llm-call-boundary.md)/[ADR-0039](0039-same-provider-reasoning-replay.md) reasoning-replay boundary — so they warrant a recorded decision rather than an implementer's guess.

## Decision

**We record the media *egress* contract for 1.AF as the following pinned choices, as an amendment to (not a supersession of) [ADR-0031](0031-llm-seam-shape-amendment-multimodal-io.md)** — ADR-0031's seam shape is unchanged; this resolves its sidecar-ownership contradiction, supplies the byte path its SSRF guardrail assumed, lands the mechanism half, and pins the resolution split. The storage substrate it consumes is [ADR-0042](0042-engine-media-storage-substrate-mediastore-deinline-retention.md).

### 1. The engine resolves handles before egress; adapters stay pure `string → string` and never fetch

The **engine** calls `MediaStore.resolveForEgress(handle, provider)` to turn a durable handle into an in-flight `MediaSource` (base64 under the inline ceiling, or — for an over-ceiling asset a provider accepts by reference — a provider-hosted ref via the sidecar below) **before** invoking the adapter. The adapter only ever sees an already-resolved in-flight source and maps it to the provider wire shape (`string → string`). An adapter **never holds a `MediaStore`, never fetches a URL, and never performs SSRF** — resolving the ADR-0031 ambiguity in favor of its body text. The 1.AE accept-but-skip comments are wired to expect a pre-resolved source. *(Considered: the adapter calling `resolveForEgress` itself — rejected: it would put a host `MediaStore` dependency and the byte path inside the seam-pure adapter, breaking I1/I2.)*

### 2. A distinct, host-injected, bytes-shaped media-egress capability

We add a **separate media-egress capability** (a host-injected port, e.g. `fetchMediaBytes`) rather than overloading `EgressCapability.body` to bytes:

- It returns a **streamed byte body bounded by a configurable maximum media-download size** — a per-fetch upper bound on the streamed body (distinct from the `byteLength`-bounded `Range` of [ADR-0044](0044-media-access-governance-read-media-save-to-cost.md)), enforced as the stream is consumed so the body is **never fully buffered in process** and an over-size response is aborted. The existing text-shaped `EgressResponse` is untouched for tool/MCP egress.
- It is a **host capability** — the platform-pure engine never opens a socket; the host (filesystem on CLI/VS Code, Rust on desktop) performs the I/O, exactly as ADR-0018 delegates LLM egress and ADR-0036 injects the transport. The engine supplies the **policy** (the SSRF primitive + the size bound); the host performs the **validated connect**.
- *Considered:* a `bytes` overload on `EgressCapability` — *rejected*: it widens the general egress contract (every host re-implements a bytes mode for a media-only need) and invites a text caller to mis-handle a byte body; a distinct, single-purpose media capability is narrower and auditable.

### 3. The SSRF mechanism half lands in 1.AF, in exactly one place, on the one shared primitive

Media-URL SSRF has a **single decision point** rather than a scatter of fetch sites, and reuses the **one shared 1.AE policy primitive** — **never a second hand-rolled parser**. The split, concretely: **(1) engine** — `deInlineMedia` detects a `url` source and calls the host `fetchMediaBytes` (it owns the *decision* to fetch — a provider-returned output URL re-hosted to a handle, an input URL re-hosted at ingest); **(2) host** — performs the validated I/O: resolve DNS, **connect by the validated IP**, and on **every redirect hop** re-run the shared primitive (`isPrivateOrLocalHost`/`extractHttpsHost`, [`@relavium/shared`](../../packages/shared/src/content.ts)) on the resolved target, so a DNS-rebind or a redirect-to-private is blocked mid-fetch, not only at the construction-time literal check; **(3) engine** — `MediaStore.put`s the returned bytes and replaces the `url` with the handle. The adapter never fetches — **no second media-fetch site**. The behavior goes **live in 1.AF** (the `MEDIA_URL_SOURCE_ENABLED` *acceptance* flag is already on from 1.AE — this lands the *mechanism*, not a flag flip). **A failsafe is pinned:** an un-rehosted `url` in a durable position **hard-fails** the handle-only durable parse (`persistableMediaRefine` rejects `url`, `content.ts`), so a fetch that is skipped or fails can never leak a live URL into an event/checkpoint. *(Considered: keep the mechanism deferred and ship output-rehost OFF — the maintainer rejected this for 1.AF in favor of closing the obligation; recorded so the docs' "Phase 2" vs "1.AF" contradiction resolves to "lands at 1.AF.")*

### 4. The `FallbackChain` owns a process-scoped sidecar and re-materializes via a narrow injected hook — never a `MediaStore` dependency

We resolve the ADR-0031 contradiction decisively:

- **The `(provider, sha256)` sidecar is held per-`FallbackChain` run-instance** (one chain per node execution, mirroring the per-run reasoning-signature handling at `beginEntry`/`stripReasoningParts`, `fallback-chain.ts`) and owned by the chain — **not** the stateless adapter, and **not** process-global (a process-wide map would leak refs across unrelated runs). This **sharpens [ADR-0031](0031-llm-seam-shape-amendment-multimodal-io.md)'s loose "process-scoped adapter sidecar" wording**. It is **never persisted, logged, checkpointed, or placed on a `ContentPart`** (asserted by a test) — the same ephemeral discipline as the ADR-0030/0039 reasoning signatures.
- **The chain stays byte-free and platform-free.** It obtains re-materialization through a **narrow resolve hook injected on `FallbackChainOptions`** — a plain function (e.g. `(handle, provider) => Promise<MediaSource>`), **not** a `MediaStore` type. The host (via the engine/`AgentRunner`) supplies the hook backed by `MediaStore.resolveForEgress`; the chain calls a function and never imports a store. *(Considered: inject a `MediaStore` directly into the chain — rejected: a host-shaped dependency in the seam-pure runner; **move strip+re-materialize entirely into the `AgentRunner`** around the chain — rejected: it splits the failover-strip logic away from the chain that owns provider advancement and duplicates the ADR-0030 `beginEntry` precedent the chain already implements.)*
- **Ordering on cross-provider advance** mirrors ADR-0030 `stripReasoningParts`/`beginEntry`: at the `beginEntry` boundary, **strip** the previous provider's ref, **re-materialize** from the canonical handle via the hook, and **await the re-upload before the retried request is sent** — the request waits on re-materialization, never races a half-uploaded ref, and the next provider's request carries no leftover foreign ref. A **same-provider retry is a no-op** (the ref is still valid). The cross-provider **context transfer is de-inlined** (B1) at the [ADR-0042](0042-engine-media-storage-substrate-mediastore-deinline-retention.md) choke point as it crosses into engine state.
- **The re-upload failure arm** is classified as **retryable-advance**: a failed re-materialization advances to the next provider (it does not fatal the turn), with a **secret-free message** that names a provider/reason and **never** a handle ref, a resolved path, a host stack, or bytes. **When there is no next provider** (a single-provider chain, or the last entry), it is **turn-fatal**, surfaced as the chain-exhausted `LlmError` — never a silent drop or a flatten-to-text.
- **On resume after a crash/restart the sidecar is empty** (it is never persisted): handles in the durable transcript are re-resolved via the hook to the resuming provider's format **before** the next request is sent — the same reconstruct-from-the-canonical-handle path [ADR-0031](0031-llm-seam-shape-amendment-multimodal-io.md) names for resume/failover.

## Consequences

### Positive

- The ADR-0031 sidecar-ownership contradiction is resolved in writing: the chain owns a process-scoped sidecar, re-materializes via a function hook, and stays byte-free/platform-free — so failover never leaks a foreign provider ref and never races a half-uploaded one, with the failure arm and ordering pinned.
- The byte path the SSRF guardrail always assumed now exists as a narrow, host-delegated, streamed, size-bounded media-egress capability — without widening the general tool-egress contract.
- The SSRF obligation is **closed in 1.AF**, in one place, on the one shared primitive, with the mechanism half (DNS-resolve + connect-by-validated-IP + per-hop redirect) on the host — and a hard-fail-on-un-rehosted-url backstop means a skipped/failed fetch can never leak a live URL durably.
- Adapters stay pure `string → string` with the resolution split pinned to the engine — I1/I2 hold, and the 1.AE accept-but-skip comments resolve to a single, audited engine resolution point.

### Negative

- Landing the SSRF **mechanism** half in 1.AF (rather than deferring it) adds a new security-sensitive network path to 1.AF and **requires a dedicated security-review pass** over the media-egress capability and the redirect re-validation (mitigated: the policy lives in the already-reviewed one shared primitive; the mechanism is host-delegated and single-purpose; an un-rehosted url hard-fails the durable parse).
- A new host capability (`fetchMediaBytes`) is per-host wiring the CLI/VS Code filesystem host and the desktop Rust host must each implement consistently (mitigated: the engine-side contract — streamed, size-bounded, SSRF-validated connect — is pinned here once; desktop already has the Rust egress + CAS pattern).
- The chain grows a process-scoped sidecar + a resolve hook on its options — more chain state to test (mitigated: it mirrors the proven ADR-0030/0039 reasoning-signature strip/replay; same-provider no-op and cross-provider re-upload-before-retry are directly tested).

### Neutral

- Per ADR-0009's append-only rule, ADR-0031 is unchanged in history; this ADR is the authoritative record for the media-egress byte path, the failover re-materialization contract, and the SSRF mechanism placement. Future seam-egress readers should read ADR-0031 **and** this.
- `width`/`height` stay **excluded** from the media metadata (ADR-0031 Y3, a pure-render concern) — no egress or failover consumer needs them, so this ADR inherits that exclusion unchanged.
