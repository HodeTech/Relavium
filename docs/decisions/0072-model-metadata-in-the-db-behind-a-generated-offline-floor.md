# ADR-0072: Model metadata in the DB behind the offline snapshot floor

- **Status**: Accepted
- **Date**: 2026-07-14
- **Supersedes (partial)**: [ADR-0064](0064-live-model-catalog.md) §4's "registry pricing is never seeded into the DB" prohibition — a models.dev **mirror** in a new table is now permitted, but its *reason* (no second drift-prone pricing **authority**) is preserved: the mirror is inert for shipped ids and `model_catalog`'s cost columns stay reserved for `source='user'`.
- **Amends**: [ADR-0071](0071-models-dev-as-the-model-metadata-source.md) §3 ("the floor" — redefined as a generated artifact *under* a DB that replaces the file cache), §4 ("additive-only" — extended to the DB write boundary via the shared admission gate), §9 ("the money guards" — the CI human gate stays at snapshot-regen; DB writes are additive-only by construction).
- **Related**: [ADR-0065](0065-provider-economics-and-extensibility.md) (the `source='user'` pricing tier + custom `base_url` id-reuse this must not disturb) · [ADR-0028](0028-workflow-resource-governance.md) (the cost cap the offline floor exists to keep working) · [ADR-0066](0066-normalized-reasoning-effort-control.md) (the per-model reasoning *capability* whose tier/effort data this stores) · [ADR-0070](0070-durable-per-model-session-cost-attribution.md) (`session_costs`, the sixth `model_catalog.id` FK referrer) · [ADR-0050](0050-cli-history-db-at-rest-posture.md) (the `history.db` the new table joins) · [ADR-0006](0006-os-keychain-for-api-keys.md) (keys stay in the keychain; a model list is non-secret). Framed by [architectural-principles.md §3](../standards/architectural-principles.md) (local-first — LLM calls go from the user's machine straight to the provider) and [ADR-0071](0071-models-dev-as-the-model-metadata-source.md) §4 (third-party catalog egress is **default-OFF**, opt-in by explicit consent).
- **Canonical homes**: the `model_metadata` / `catalog_meta` DDL → [database-schema.md](../reference/shared-core/database-schema.md); the seam signature → [llm-provider-seam.md](../reference/shared-core/llm-provider-seam.md); the commands → [commands.md](../reference/cli/commands.md); versions → [tech-stack.md](../tech-stack.md).

## Context

Model data lives in **two physically separate stores that merge only at read time**, and one of them is compiled into the binary:

1. **Provider availability** — which models a user's API key can reach — is written to the SQLite `model_catalog` table ([ADR-0064](0064-live-model-catalog.md)), rows discriminated by `source` (`static`|`live`|`user`), keyed `(provider_id, model_id)`, whose `id` is an FK target from **six** tables (`agents`, `stepExecutions`, `runCosts`, `agentSessions`, `sessionMessages`, and — since [ADR-0070](0070-durable-per-model-session-cost-attribution.md) — `sessionCosts`). Its cost columns are reserved for `source='user'` pricing only — ADR-0064 §4 **forbids** seeding the generated registry's pricing there ("a second, drift-prone home").
2. **models.dev metadata** — price, limits, reasoning shape + effort values, request capabilities ([ADR-0071](0071-models-dev-as-the-model-metadata-source.md)) — lives **entirely outside the DB**: a generated `snapshot.ts` shipped in the binary, an optional `~/.relavium` raw file cache, and an in-memory `refreshed` overlay, read through the single pure seam `catalogModel(id) = refreshed[id] ?? CATALOG_SNAPSHOT[id]`.

The maintainer wants the DB to become the authoritative, **regularly-refreshed** home for the full models.dev field set — adding input/output modalities, `tool_call`, knowledge cutoff, description, and `structured_output` on top of what [ADR-0071](0071-models-dev-as-the-model-metadata-source.md) already carries — plus a per-model **visibility** flag for a future `/settings > /models` "which models show in the picker" toggle. `/models` + `Ctrl+R` should refresh **both** axes (provider lists *and* models.dev) and persist. The stated goal is "**no static-in-binary data**."

The stakes of getting this wrong are a **money-safety regression**, which is why "no static-in-binary" cannot be taken literally for the pricing surface. The shipped snapshot is not incidental static data — it is the **offline floor of a safety control**: [ADR-0028](0028-workflow-resource-governance.md)'s cost cap must price every model **offline, on first run, with no network** ([ADR-0071 §3](0071-models-dev-as-the-model-metadata-source.md)). A DB-only catalog with an empty first-run DB is the *missing-price* case product-wide.

The honest asymmetry is that **a wrong price still engages the cap; a missing price does not**: the governor degrades an unpriceable model to `allow`, or `strict_cost_cap` rejects every round on that model. Dropping the floor defeats the very control it protects.

Two further constraints frame the design. `packages/core` (the engine) has **zero platform imports**, and its pricing path resolves through `catalogModel` only via `@relavium/llm` helpers (`estimateMaxNextCost` and friends) — so the DB must stay host-side and the seam must stay pure ([architectural-principles.md](../standards/architectural-principles.md)). And third-party egress to models.dev is **default-OFF** ([ADR-0071 §4](0071-models-dev-as-the-model-metadata-source.md)); the availability axis is first-party (the user's own key to the user's own provider), so the two refresh axes must keep different auto postures.

## Decision

**We will store the full models.dev field set in a new DB table and keep the generated snapshot as the offline money-and-wire floor *under* the DB — the DB is the live, refreshed source, and the binary snapshot is the terminal fallback that guarantees the cost cap works offline.** "No static-in-binary" is reframed as *retire the hand-typed `MODEL_PRICING` (done in ADR-0071) and the `~/.relavium` file cache — not the generated safety floor.*

1. **A new sibling table `model_metadata`, keyed by canonical `model_id` alone** (matching `CatalogSnapshot = Record<string, CatalogModel>`, whose `CatalogModel` carries its own `provider` field), holds the models.dev metadata — **not** new columns on `model_catalog`. *Considered columns on `model_catalog` (rejected: models.dev metadata is provider-agnostic and would be duplicated across every provider row of a shared id, and would collide with the 3-value `source` enum and the six-FK rebuild constraint).* Its full DDL is a spec with one home in [database-schema.md](../reference/shared-core/database-schema.md), not restated here.

2. **The generated `snapshot.ts` stays compiled into the binary as the offline floor.** The pure read seam `catalogModel(id) = refreshed[id] ?? CATALOG_SNAPSHOT[id]` is **byte-unchanged** (only `installCatalogRefresh`, which builds `refreshed`, is refactored — see point 3). On a fresh/offline/never-refreshed install — or an empty, torn, or older-schema DB — the host installs nothing, `refreshed` stays `{}`, and `catalogModel` falls to the snapshot: all shipped models priced offline, zero DB reads. **The DB is never the terminal money source; it only *backs* the overlay, exactly as the file cache does today.** *Considered a DB-only design with no floor (rejected: a first-run/offline/torn-DB window with no cost cap defeats ADR-0028).*

3. **A single pure `admitRefreshedModels(models)` gate is the additive-admission rule, shared by both the overlay install and the DB writer. It lives in `@relavium/llm`, is typed over `CatalogModel` (never a Drizzle row type), and is imported by the host DB writer** — so the dependency direction stays `apps/cli → @relavium/llm`, never `@relavium/llm → @relavium/db`, and engine purity holds. The gate encodes three money invariants, each with an explicit test:
   - **(a) Snapshot-membership exclusion keys on compile-time `CATALOG_SNAPSHOT[id]`, never the mutable DB `origin` column.** A shipped id is rejected because it is *in the snapshot* (`CATALOG_SNAPSHOT[id] !== undefined`, today's `lookup.ts`), not because a DB row happens to say `origin='shipped'`. This is load-bearing: if a long-tail `gpt-6-mini` enters the DB `origin='refreshed'` at 10/30, and a later binary promotes it into the snapshot at a human-approved 50/150, a projection that read `WHERE origin='refreshed'` (before the SHA-gated reseed flips origin) would let the stale 10/30 row shadow the snapshot via `refreshed[id] ??` and undercharge the cap. The read-path exclusion **must** be snapshot membership; `origin` is bookkeeping.
   - **(b) A new id is admitted only if both base prices are finite and positive** (`Number.isFinite(input) && input > 0 && Number.isFinite(output) && output > 0`). `NaN <= 0` is `false`, so a `NaN`-coerced money field must be caught by the finite check, not the `> 0` check — otherwise a torn row passes the gate and `NaN` propagates into the governor. This keeps the fail-closed promise concrete: a row with a missing/non-finite required price is **omitted** (falls to the snapshot), never admitted.
   - **(c) Money-column NULLness survives the DB round-trip.** `cache_read`/`cache_write` are NULLABLE; the DB→`CatalogModel` projection maps NULL → **`undefined`, never `0`** ([ADR-0071 §10](0071-models-dev-as-the-model-metadata-source.md): `0` means "no discount" and would price a cached read at zero — a silent undercharge). Base `input`/`output` are NOT NULL and gate admission (b).

   Because the gate is one shared pure function, a background or `Ctrl+R` refresh cannot lower a shipped price or admit an unpriceable long-tail row on the overlay path *by construction*; on the DB path the same guarantee holds **for a correctly-wired writer that calls the same gate**, enforced by a parity test (not by construction — see Negative). Because a refresh never touches a shipped id, the [§9](0071-models-dev-as-the-model-metadata-source.md) human money-review gate legitimately **stays at snapshot-regen time**; long-tail price movement in the DB is additive drift (see Negative for why that is accepted, not a loophole).

4. **`model_catalog` gains one additive `visible` boolean flag**, orthogonal to `is_active`: a reachable-but-hidden model is `is_active=1, visible=0`. **Every write path that rewrites a `model_catalog` row must preserve `visible` by read-modify-write** — the per-model `upsert` already uses the `input ?? existing ?? default` idiom, and the bulk `replaceProviderModels` (which defaults unlisted columns rather than carrying them) must explicitly re-read and re-write `visible` so a routine availability refresh never silently un-hides a model the user hid. `visible` is **identity/preference**, a hard filter *above* ADR-0064 §6's "dim, never hide" availability rule.

5. **Shipped models keep their money and wire-capability data pinned to the reviewed snapshot; only *pure* enrichment auto-refreshes.** `catalogModel` stays byte-unchanged, so a shipped model's **price and its 400-avoidance wire fields** (reasoning shape, effort values, `request_capabilities`) move **only by a PR-reviewed binary release**. **Pure enrichment** (modalities, knowledge cutoff, description) is DB-only and freely auto-refreshed. Because `admitRefreshedModels` *rejects* shipped ids (point 3a), enrichment of a shipped row is a **separate write path** that must touch **only an allow-list of pure-enrichment columns** — a shipped row's money and wire columns are written **once**, at the SHA-gated boot seed (point 6), and never by a refresh UPSERT. *Considered the aggressive split — merging shipped reasoning/capabilities from the DB in the hot read path (rejected: it adds merge risk to a safety-critical path and removes shipped wire capabilities from human review — the value that avoids a live 400).*

6. **The DB mirrors *all* models — shipped rows seeded `origin='shipped'`, the long tail `origin='refreshed'`** (seed shape "Option B") — so any model browser reads a uniform field set from one home. **A singleton `catalog_meta` row is the durable cursor**: it holds the seeded-snapshot SHA (which gates re-seeding after a binary upgrade), the per-axis TTL timestamps, and a `catalog_schema_version` **checked before admission** so a row written by an older normalizer is treated as absent (falls to the snapshot) until a refresh rewrites it. The seed is written **host-side at boot from `CATALOG_SNAPSHOT`** (idempotent, SHA-gated), never in a migration `.sql` — money never lands in a frozen SQL artifact that would itself become a §9 money surface. *Considered Option A — the DB holds only the refreshed long tail (rejected: it splits the browser's read across two homes; Option B's only cost is the SHA-reconcile-on-upgrade step).*

7. **The DB replaces the `~/.relavium` file cache as the durable overlay backing, and the host installs from exactly one authoritative backing at a time**: the DB when it is non-empty **and** at the current `catalog_schema_version`, otherwise the file cache (retired only in the final phase, once the DB path is proven). This closes the wholesale-replace hazard — an empty or torn DB must **not** overwrite a good file-cache overlay. `Ctrl+R` / `models refresh` run **both** axes on two independent TTL cursors with different auto postures: **availability default-ON** (first-party — the user's own key), **catalog default-OFF** (third-party egress). **Opening the `/models` picker runs only the availability axis**; the models.dev axis fetches **only** on explicit `Ctrl+R` / `models refresh`, or when `[catalog] auto_refresh = true` — never bound to picker-open, so opening a picker is not silent third-party egress. The 20-second fetch happens **outside** any write transaction; the persist is a short separate `BEGIN IMMEDIATE` + busy-retry write ([ADR-0064](0064-live-model-catalog.md) 2.5.I).

**Relationship to the amended ADRs** (recorded append-only there — a dated blockquote note is appended to each; their Status stays Accepted): the ADR-0064 §4 change is a **partial supersession** (a prohibition, "never seed pricing into the DB", becomes a scoped permission), matching ADR-0071's vocabulary for its own partial-clause reversals; the ADR-0071 §3/§4/§9 changes are **amendments** (the floor still exists, refined; additive-only extended, not reversed).

## Consequences

### Positive

- **The model list and full metadata stay current in one queryable home** (the DB), auto-refreshed from both sources, instead of a binary snapshot that only moves on release — the maintainer's core ask.
- **The offline cost cap is preserved byte-for-byte** (`catalogModel` and the snapshot data unchanged; the snapshot is the terminal answer), and the runtime money guard is *strengthened*: one shared `admitRefreshedModels` with a snapshot-membership exclusion, a finite-and-positive price check, and NULL→`undefined` money projection.
- **The engine stays platform-pure** — `packages/core` and pure `@relavium/llm` never import `@relavium/db`; the gate lives in `@relavium/llm` typed over `CatalogModel`, the host reads the DB and injects plain rows via the existing `installCatalogRefresh` setter, the same seam the file cache uses today.
- **Per-model visibility** becomes a first-class, refresh-safe preference, unblocking `/settings > /models`.
- **Every migration is additive and reversible** (an `ADD COLUMN` with a constant default; a fresh `CREATE TABLE`), so the FK graph and the picker are never rebuilt.

### Negative

- **Floor erosion is the fatal risk** — if a later phase ever made the DB the *terminal* price source (dropped the snapshot), a fresh/torn/mid-migration DB would return `undefined` for every id and the cap would degrade product-wide. *Mitigation: this ADR forbids the DB from ever being the money authority; the snapshot stays the floor; a torn projection fails **closed** (omit the row → fall to the floor), and the read-path exclusion keys on compile-time snapshot membership (point 3a), not the mutable `origin`. Load-bearing; explicitly tested.*
- **Long-tail prices have no human baseline, so a large drop is admitted, not blocked.** `admitRefreshedModels` is stateless: a long-tail id admitted at $2.00 is re-admitted if a later refresh reports $0.02 (still `> 0`), because the §9 CI gate covers only shipped ids. *This is accepted, deliberately.* For a long-tail model, models.dev is the **only** price source — there is no human-verified baseline to regress from, unlike a shipped model. Keeping the old higher price would **over**charge (the conservative cap direction, but a false positive that blocks legitimate work on a real provider price cut), and a stateful magnitude-guard would put state into a deliberately-pure gate and add exactly that false-positive mode. This is also the **already-accepted ADR-0071 posture** (the DB does not weaken it); the money-critical set — the shipped snapshot — is fully protected; and the direction that matters (a too-low price) is bounded for shipped ids by the human gate.
- **Provider-agnostic keying meets ADR-0065's id reuse.** Because `model_metadata` and `catalogModel(id)` key on `model_id` alone, a custom `base_url` endpoint (which reuses an existing provider id, [ADR-0065](0065-provider-economics-and-extensibility.md) §6) whose `model_id` string-collides with a real models.dev **long-tail** id would inherit that id's enrichment. *Mitigation: this collision already exists today for the snapshot (same key space), and is bounded the same way — the collision can only carry **pure enrichment** into the overlay; **price** still flows through `admitRefreshedModels` and the user's explicit `source='user'` pricing wins (ADR-0065). Re-keying `model_metadata` to `(provider, model_id)` was rejected: it would break the byte-identical `catalogModel` seam the whole floor rests on.*
- **The DB-path additive guarantee needs a parity test, not construction** — nothing structurally forces the DB writer to call `admitRefreshedModels`; a re-implementation could admit a different set. *Mitigation: the gate is a single pure export both callers use, and a parity test asserts the DB writer and the overlay install admit the identical set from the same input.*
- **Shipped-row enrichment is a second, un-gated write path** (the gate rejects shipped ids, points 3a/5). *Mitigation: that path writes **only** allow-list enrichment columns; a test asserts a shipped row's money/wire columns are never mutated by a refresh and its DB money is never read for pricing.*
- **Frozen normalizer in stored rows** — normalized rows freeze today's normalizer. *Mitigation: `catalog_schema_version` in `catalog_meta`, checked before admission; older-version rows are treated as absent until rewritten, and the file cache is kept until the version gate is proven.*
- **`description` may be `NULL` on arrival** — models.dev publishes no per-model description today. *The column is carried at the maintainer's explicit request; it is NULLABLE and populated only when a Relavium-owned overlay (like `deprecations.ts`) or a second source exists — pure enrichment, never on the pricing/wire path.*
- **Boot latency** — a DB open on every invocation would slow `--help`/`--version` (today `seedCatalog()` runs unconditionally but opens no DB). *Mitigation: the DB seed/read is gated so it runs only after the resolved command is one that prices; cheap paths stay on the no-DB path. The exact command-classification mechanism is an implementation detail deferred to the read-cutover phase.*
