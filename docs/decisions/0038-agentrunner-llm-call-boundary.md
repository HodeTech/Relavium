## Context

Workstream **1.O** builds the `AgentRunner` ‚Äî the per-node LLM execution that fills the `NodeExecutor` seam ([ADR-0036](0036-run-loop-substrate-event-bus-and-execution-host.md)) for `agent` vertices: assemble the prompt, call `@relavium/llm` through the seam, re-emit the streaming `agent:*` events, run the tool-call loop through the `ToolRegistry` ([ADR-0037](0037-engine-tool-execution-boundary.md)), feed usage to a cost tracker, and map the outcome to one `NodeOutcome`. It is the **join** on the engine critical path ‚Äî its three dependencies (1.K `FallbackChain`, 1.N run loop, 1.T `ToolRegistry`) are all merged.

The run loop hands a node executor a `NodeExecContext` ([node-executor.ts](../../packages/core/src/engine/node-executor.ts):96-109) carrying `vertex` / `runOutputs` / `inputs` / `emit` / `signal` / `attemptNumber` ‚Äî and **nothing else**. It deliberately does **not** carry a provider, a credential, or a cost tracker: the loop is provider-agnostic by construction. So the `AgentRunner` needs its **own injected dependency boundary** to obtain a concrete `LlmProvider` and resolve credentials ‚Äî and that boundary is named in **no** Accepted ADR. [ADR-0036](0036-run-loop-substrate-event-bus-and-execution-host.md) pins the `ExecutionHost` (clock / ids / store / abort); [ADR-0037](0037-engine-tool-execution-boundary.md) pins the `ToolHost` (the side-effecting I/O tools need). **Neither covers the LLM call** ‚Äî provider-instance resolution and the credential injection for the in-core agent turn.

Five facts (verified against the merged seam) frame the decision, and getting it wrong trips three non-negotiables at once (rule 4: no vendor type across the `@relavium/llm` seam; rule 5: `@relavium/core` has zero platform imports; rule 6: keys live in the keychain, never in plaintext / logs / the frontend / a job payload):

1. **The authored provider is an id string; the chain wants an instance.** `Agent.provider` and `fallback_chain[].provider` are provider-id strings (`ProviderSchema`, the closed seam set ‚Äî [agent.ts](../../packages/shared/src/agent.ts):127,140), but `FallbackPlanEntry.provider` is a concrete `LlmProvider` **instance** ([fallback-chain.ts](../../packages/llm/src/fallback-chain.ts):62). The engine **cannot** construct an adapter instance: the adapters import the vendor SDKs and `@types/node`, which the ESLint engine fence and `tsconfig.purity.json` (`types: []`) forbid in `packages/core`. A provider-id ‚Üí instance resolver must be supplied by the host. **This is the one genuinely-new capability 1.O needs.**
2. **The credential / timer seam already exists ‚Äî on the chain.** `FallbackChainOptions` already declares `keyFor: (provider: ProviderId) => string | Promise<string>` (documented host-aware), `sleep`, `now`, and `onAuthError`, plus `costTracker` and `onAttempt` ([fallback-chain.ts](../../packages/llm/src/fallback-chain.ts):106-140). 1.O must **not** re-declare the credential/timer fields in a parallel seam ‚Äî that would create a second home for the key resolver (a one-canonical-home violation and the P0 key-mis-wiring surface).
3. **Two pieces of chain state have *different* lifetimes ‚Äî and one is not what a na√Øve reading assumes.** `#cooldownUntil` ([fallback-chain.ts](../../packages/llm/src/fallback-chain.ts):212) and `#authRefreshed` (:214) are **`FallbackChain` instance** fields ‚Äî they survive across `generate()`/`stream()` calls, so reusing one chain across a node's tool-loop continuations preserves cooldown and the single auth-refresh. **But `#lastProvider` is a `ChainRun` field (:581), and a fresh `ChainRun` is created on *every* `generate()`/`stream()` call (:249,:308)** ‚Äî so the cross-provider reasoning strip-latch **resets every call regardless of chain reuse**. Reusing the chain is necessary (for cooldown) but **not sufficient** for reasoning-replay correctness across continuations; that gap is closed by [ADR-0039](0039-same-provider-reasoning-replay.md)'s instance-scoped-latch change-set, on which this ADR depends.
4. **The credential string transits the engine at the call site.** `LlmProvider.generate(req, key: string)` / `stream(req, key: string)` accept the credential directly ([types.ts](../../packages/llm/src/types.ts):457-464). Because `@relavium/core` constructs and runs the `FallbackChain`, the resolved credential string passes through engine memory transiently on its way to the adapter. So the guarantee is **not** "no key crosses the seam"; it is the rule-6 discipline (below).
5. **The executor is shared across concurrent runs.** `WorkflowEngine` injects one `#executor` and creates a `RunExecution` per run that holds the same instance ([engine.ts](../../packages/core/src/engine/engine.ts):131,772); the run loop owns the run-wide `cumulativeCostMicrocents` (:686, "a per-node executor cannot know it"). So any **mutable** state the AgentRunner holds (a `CostTracker`, a chain) must be scoped **per node execution**, never per node definition or per executor instance, or concurrent runs mix cost/failover state.

Two adjacent constraints also bind 1.O's structure: [ADR-0028](0028-workflow-resource-governance.md) mandates a **pre-egress** budget check (filled by 1.AC, but 1.O must leave the insertion point), and [ADR-0024](0024-agent-first-entry-point-agentsession.md)/[0025](0025-agent-surface-refines-desktop-scope.md)/[0026](0026-session-export-to-workflow.md) mandate that `AgentSession` (1.V) **reuses** the AgentRunner rather than forking the substrate.

## Decision

**We will define the AgentRunner's LLM-call boundary as the *minimal* extension of the existing seams: the host injects platform capabilities only ‚Äî a provider-id ‚Üí instance resolver plus the chain's timing/credential capabilities ‚Äî and the AgentRunner owns everything stateful and engine-side (the per-node-execution chain, the cost tracking, the event emission). No key is stored, inspected, logged, persisted, or sent to the frontend by `packages/core`.** This is the LLM-call sibling of [ADR-0037](0037-engine-tool-execution-boundary.md)'s `ToolHost`: the engine owns the *policy* (assembling the plan from the authored config it already holds in the `PlanVertex`, sequencing the call, classifying the outcome, emitting events), the host supplies the *mechanism* (the concrete adapter and the keychain credential, attached at the call boundary).

**The injected `AgentRunnerDeps` carries platform capabilities only:**

- **`resolveProvider(providerId: ProviderId): LlmProvider | undefined`** ‚Äî the one genuinely-new capability, mapping the authored provider-id strings to the concrete adapter instances `FallbackPlanEntry.provider` needs, so `packages/core` never imports an adapter. It returns `undefined` for an id the host did not wire; the AgentRunner maps that to a typed `NodeFailure` with code **`internal`** (a host-wiring gap, not an authoring error ‚Äî distinct from the absent-`resolvedAgent` case below). On the desktop these adapters route egress through Rust ([ADR-0018](0018-desktop-execution-and-rust-egress.md)); on Node they call directly; in Phase-2 cloud the relocated host supplies them.
- **The chain's host-capability fields ‚Äî `keyFor`, `sleep`, `now`, and the optional `onAuthError`** ‚Äî typed by indexed access to `FallbackChainOptions` (e.g. `keyFor: FallbackChainOptions['keyFor']`) so a change to the chain options cannot drift a second copy. The AgentRunner forwards these straight into the `FallbackChain` it builds. `onAuthError` (the single out-of-band credential refresh) is **owned by the host** and only forwarded.

**The AgentRunner ‚Äî not the host ‚Äî owns the cost path** (fact 5): it creates a `CostTracker` **per node execution** and supplies its **own** `onAttempt` that converts each `AttemptRecord` into a `cost:updated` `NodeStreamEvent` (deriving `attemptNumber` by counting **non-skipped** attempts, not the positional index). The host never provides `costTracker` or `onAttempt` (a shared tracker would mix concurrent-run state; emitting governance events is the executor's job, not a host callback). The clean ownership split: **host** = provider + credential + clock/delay; **AgentRunner** = per-node-execution chain + tracker + `onAttempt`‚Üíevent; **run loop** = the single authority for the run-wide cumulative cost.

**Credential discipline (fact 4, the accurate guarantee):** `packages/core` never **stores, inspects, logs, persists, or forwards to the frontend / into a job payload** the credential; it is threaded **opaquely** through the in-process chain and resolved at the `@relavium/llm` call boundary. `keyFor` returns whatever the surface's credential model uses ‚Äî a keychain-backed reference resolved inside the Rust adapter on the desktop, a managed token in managed mode ([ADR-0012](0012-managed-inference-dual-mode.md)/[0013](0013-managed-key-vault-and-pools.md)), or the raw key on a Node BYOK surface ‚Äî and the value never appears in an `agent:*`/`node:failed`/`run:failed` event, a log line, or a checkpoint. This is the rule-6 discipline ([ADR-0006](0006-os-keychain-for-api-keys.md)/[ADR-0018](0018-desktop-execution-and-rust-egress.md)), not a "no key crosses the seam" claim, which the `generate(req, key)` contract makes false.

**The per-node-execution `FallbackChain`:** 1.O constructs **one `FallbackChain` per node execution** (per run/dispatch ‚Äî never per node definition, fact 5), reused across that node's tool-loop continuations so per-provider cooldown and the [ADR-0039](0039-same-provider-reasoning-replay.md) instance-scoped strip-latch survive the loop, and discarded when the node settles. It builds the `FallbackPlanEntry[]` from the resolved agent + node config (the primary `{provider, model, maxAttempts}` plus each `fallback_chain` entry, provider ids run through `resolveProvider`), node overrides winning over the agent default (model / temperature / max_tokens / output_schema); `tools:` **narrows** the agent's grant and never widens ([ADR-0029](0029-tool-policy-hardening.md), parser-enforced ‚Äî a node listing a tool the agent lacks fails validation).

**A pure, always-pass pre-egress hook** runs before each **tool-loop turn's** seam call (`chain.stream`) ‚Äî an **engine-side** call site (not a host capability; the budget governor is engine-side per [ADR-0028](0028-workflow-resource-governance.md)) whose 1.O implementation is a **no-op that always permits**. Its behavioral contract in 1.O is "always pass," so `budget_exceeded` cannot fire until 1.AC fills the gate. **1.AC owns the budget estimator.** Because a `FallbackChain` makes *several* provider egresses per turn (primary + retries + fallbacks), the precise *per-attempt* budget gate ‚Äî immediately before each provider egress ‚Äî is a `FallbackChain` pre-attempt hook **1.AC adds**; the turn-boundary hook 1.O leaves is the coarse insertion point, not the final per-egress one. 1.O emits **no** `budget:*` / `run_timeout` events ‚Äî those are run-level and are not in the `NodeExecutor` in-node event set ([node-executor.ts](../../packages/core/src/engine/node-executor.ts):34-39).

**Structure (mandated by adjacent ADRs, recorded here so 1.O builds it right the first time):**

- 1.O ships the **single dispatching `NodeExecutor`** the engine holds: `execute(ctx)` switches on `ctx.vertex.type` (the `EngineNodeType` axis); the `agent` arm narrows `ctx.vertex.config` via `config.kind === 'agent'` to `AgentPlanConfig` ([run-plan.ts](../../packages/core/src/run-plan.ts):74-82) and runs the AgentRunner; every non-agent type returns a **loud, typed `failed`** stub until 1.P (never a silent default). An **absent `resolvedAgent`** at dispatch returns a typed `NodeFailure` (`code: 'validation'`, naming the unresolved `agent_ref`) ‚Äî an authoring/config error, distinct from the host-wiring `internal` of an unresolved provider id ‚Äî never a raw throw (the loop flattens a throw to `internal`, losing the classification).
- The per-turn LLM+tool loop is factored as a **correlation-key-agnostic turn core** wrapped by the thin `NodeExecutor` adapter ‚Äî the reuse [ADR-0024](0024-agent-first-entry-point-agentsession.md)/0025/0026 require so 1.V reuses it unchanged. The turn-core parameters are: `system` + `messages` + `tools` + the fallback plan + `emit` + `signal` + `nodeId` + the tool registry + a **`limits`** object. The **`limits`** include a **runner-default max-tool-turns cap** ‚Äî an unbounded `tool_use` loop is a cost/DoS hole with no other owner between 1.O and 1.V, so 1.O ships the default; the *authored* hard cap and the loud `turn_limit` surfacing are the 1.V session knob, wired so 1.V overrides the default without restructuring the loop. **On a tool-loop continuation, the assistant turn re-fed in `messages` carries its reasoning `ContentPart`(s)** so the same-provider signed-reasoning replay ([ADR-0039](0039-same-provider-reasoning-replay.md)) works ‚Äî this is the explicit contract point between the two ADRs. Run-path-only concerns (`run.outputs` placement, the run-correlated `agent:file_patch_proposed`) live in the **adapter**, never the core. The turn-core parameter shape is a **frozen internal contract** with a test asserting both the node adapter and a session-style caller satisfy it ‚Äî the "no 1.V rewrite" guarantee enforced by a test, not by intent.

Considered alternatives:

- **A full `AgentRunnerDeps { resolveProvider, keyFor, sleep, costTracker, onAttempt }` that re-declares or embeds the whole `FallbackChainOptions`** (rejected) ‚Äî re-declaring `keyFor`/`sleep` creates two homes for the credential seam (drift + the P0 of logging the wrong one); embedding the *whole* `FallbackChainOptions` would let the host inject `costTracker`/`onAttempt`, which fact 5 forbids (shared mutable state across concurrent runs; the executor must own cost emission). The bundle carries only the host-capability subset, typed by indexed access, and the AgentRunner owns the cost path.
- **The AgentRunner imports `withFallback` and resolves keys itself** (rejected) ‚Äî needs keychain / platform code in `packages/core`, breaking rules 5/6.
- **The host pre-assembles the full `FallbackPlanEntry[]` + `FallbackChainOptions` per dispatch** (rejected as primary) ‚Äî pushes the agent‚Üíplan mapping to every surface, duplicating logic the engine already owns from the `PlanVertex`. The chosen middle keeps plan assembly in the engine and injects only the provider resolver + the host capabilities.
- **Fold this into ADR-0036's `ExecutionHost` or ADR-0037's `ToolHost`** (rejected) ‚Äî the LLM call is a distinct concern with a distinct lifecycle (per-node-execution chain, provider cooldown, reasoning latch) from the run-loop substrate and the tool I/O. A sibling seam keeps each interface minimal and auditable.

This ADR pins the **boundary and the injection shape**; the exhaustive `AgentRunnerDeps` / turn-core interface lands in its canonical home (the agent-runner reference page under [docs/reference/shared-core/](../reference/shared-core/), created with 1.O ‚Äî rule 8). Two paired change-sets land **with 1.O in the same PR or behind a hard acceptance gate** (the same coupling discipline [ADR-0039](0039-same-provider-reasoning-replay.md) records), **not** silently deferred: the same-provider signed-reasoning replay ([ADR-0039](0039-same-provider-reasoning-replay.md), which also moves the strip-latch to instance scope per fact 3), and the node-side `output_schema` validation ‚Äî the seam's `responseFormat` is a **request-side hint only** (no adapter validates the response; DeepSeek degrades to bare `json_object`), so 1.O validates the returned content against `output_schema` and maps a miss to `validation` (`retryable: false`). The **failure classification** is owned by [error-handling.md](../standards/error-handling.md); the **override precedence** (a node's `output_schema` wins over the agent default) is documented in [node-types.md](../reference/shared-core/node-types.md) and [agent-runner.md](../reference/shared-core/agent-runner.md) (the homes updated with 1.O).

## Consequences

### Positive

- The engine stays **platform-free** by construction: `packages/core` imports only the `@relavium/llm` seam barrel (no adapter, no vendor type, no `node:*`), and the credential is threaded opaquely and never stored/inspected/logged/persisted by core.
- **One credential seam, no drift**: the deps forward the host-capability subset of `FallbackChainOptions` by indexed-access types rather than minting a second `keyFor`.
- **Concurrency-safe cost**: the AgentRunner owns a per-node-execution `CostTracker` + `onAttempt`, so the shared executor never mixes cost/failover state across runs; the run loop stays the cumulative authority.
- Per-provider cooldown and (with [ADR-0039](0039-same-provider-reasoning-replay.md)) the strip-latch are **correct across the tool loop** via one per-node-execution chain.
- A clean, pure **pre-egress insertion point** for 1.AC, and a **reusable turn core** so 1.V plugs in without a rewrite.

### Negative

- `AgentRunnerDeps` is a **third injected seam** the surfaces wire (alongside `ExecutionHost` and `ToolHost`) ‚Äî mitigated by a single in-memory test helper and a stub `resolveProvider`.
- 1.O's full acceptance depends on two paired change-sets ([ADR-0039](0039-same-provider-reasoning-replay.md) reasoning replay + the 1.K instance-latch move; node-side `output_schema` validation) that this ADR scopes out to keep the boundary focused ‚Äî they **must land with 1.O in the same PR or behind a hard acceptance gate**, tracked as explicit acceptance criteria, never silently deferred.
- The credential discipline is a **review-enforced invariant** (core never stores/logs/persists the credential), not a type-level one ‚Äî the `generate(req, key)` contract means a key string is reachable at the call site, so a Leakwatch pass over the diff and the secret-free-event checks remain binding.
